# Спецификация: Обновление не до последней версии

## Обзор

Добавить возможность обновлять версии в `requirements.yml` только в пределах минорной или патч-версии (semver), сохраняя текущее поведение по умолчанию (обновление до latest).

## CLI интерфейс

### Новые флаги

```
-m, --minor    Обновлять только в пределах minor-версии (та же major)
-p, --patch    Обновлять только в пределах patch-версии (та же major и minor)
```

### Взаимное исключение

Флаги `-m/--minor` и `-p/--patch` являются **взаимно исключающими**. Валидация выполняется на уровне `OptionParser` — при передаче обоих флагов выбрасывается ошибка до начала выполнения.

### Поведение по умолчанию

Запуск без флагов сохраняет текущее поведение: обновление до **последней** доступной версии (latest).

## Логика выбора версий

### Режимы работы

| Режим | Описание | Пример |
|-------|----------|--------|
| **latest** (default) | Максимальная версия из всех доступных | `1.2.3` → `2.1.5` |
| **minor** | Максимальная версия с той же `major` | `1.2.3` → `1.9.9` (но не `2.0.0`) |
| **patch** | Максимальная версия с той же `major` и `minor` | `1.2.3` → `1.2.8` (но не `1.3.0`) |

### Фильтрация версий

1. **Pre-release версии** (например, `1.2.4-alpha`, `1.2.4-beta.1`) **игнорируются**. Выбираются только стабильные (GA) версии.
2. **Не-semver теги** (например, `latest`, `stable`, `dev`) игнорируются с предупреждением.

### Обработка префиксов тегов

Поддерживаются **только теги в чистом semver формате** (`1.2.3`). Теги с префиксами (`v1.2.3`, `release-1.2.3`) игнорируются.

## Обработка ошибок и предупреждения

### Отсутствие подходящих версий

Если в режиме `minor`/`patch` не найдено подходящих версий (например, текущая `1.2.3`, а доступны только `2.0.0`, `2.1.0`):

- Выводится предупреждение в **stderr** с префиксом `Warning:`
- Версия в `requirements.yml` **не изменяется**
- **Exit code = 0** (успех)

### Не-semver теги

При обнаружении тегов, не соответствующих semver:

- Выводится предупреждение в **stderr** с префиксом `Warning:`
- Тег игнорируется при выборе версии
- Обработка продолжается

### Формат предупреждений

```
Warning: <описание проблемы>
```

Примеры:
```
Warning: no suitable versions found for 'foo' within minor version (current: 1.2.3)
Warning: skipping non-semver tag 'latest' for repository 'https://...'
```

## Вывод информации

При успешном обновлении выводится:

1. Старая версия
2. Новая версия
3. **Объяснение выбора** (почему выбрана именно эта версия)

Пример:
```
foo: 1.2.3 → 1.2.8 (max patch version for 1.2.x)
bar: 1.2.3 → 1.9.5 (max minor version for 1.x)
baz: 1.2.3 → 2.1.0 (latest)
```

## Архитектура

### Enum Versions

Модифицировать существующий enum:

```crystal
enum Versions
  Patch
  Minor
  Latest
end
```

**Удалить** вариант `Major` (эквивалентен `Latest`).

### Парсинг semver

Использовать **Crystal stdlib `Version`** без добавления внешних зависимостей. Pre-release версии определяются через наличие суффикса (например, `-alpha`, `-beta.1`).

### Структура кода

Расширить существующий модуль `Versions` методами для выбора версии в пределах `minor`/`patch`:

- `select_latest(tags : Array(String)) : String`
- `select_minor(tags : Array(String), current : Version) : String?`
- `select_patch(tags : Array(String), current : Version) : String?`

Методы возвращают `nil`, если подходящая версия не найдена.

## Тестирование

### Типы тестов

1. **Unit тесты** — логика выбора версий (границы semver уровней)
2. **Integration тесты** — с реальными git-репозиториями
3. **CLI spec тесты** — валидация аргументов

### Тестовые сценарии

| Сценарий | Режим | Ожидаемый результат |
|----------|-------|---------------------|
| Максимальная patch в пределах | `--patch` | Выбрана максимальная patch версия |
| Максимальная minor в пределах | `--minor` | Выбрана максимальная minor версия |
| Pre-release фильтрация | любой | Pre-release версии игнорируются |
| Нет подходящих версий | `--minor`/`--patch` | Warning, версия не изменена |
| Взаимно исключающие флаги | `-m -p` | Ошибка CLI |
| Не-semver теги | любой | Warning, теги игнорируются |
| Префиксы тегов не теряются | любой | Префиксы сохраняются в requirements.yml |

### Фикстуры

- **Существующие фикстуры** используются без изменений
- **Новые фикстуры** добавляются по мере необходимости (например, для pre-release тестов)

### Integration тесты

Использовать **репозиторий самого проекта** (`reqs-up`) как тестовый git-репозиторий с тегами.

## Примеры использования

```bash
# Обновление до последней версии (по умолчанию)
reqs-up

# Обновление только в пределах minor-версии
reqs-up -m
reqs-up --minor

# Обновление только в пределах patch-версии
reqs-up -p -f requirements.yaml
reqs-up --patch --file requirements.yaml

# Ошибка: взаимно исключающие флаги
reqs-up -m -p -f requirements.yml  # Error: --minor and --patch are mutually exclusive
```
